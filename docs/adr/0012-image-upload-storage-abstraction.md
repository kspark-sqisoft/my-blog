# ADR-0012: 이미지 업로드를 저장소 추상화로 구현 (로컬 저장 → S3 확장)

## 상태 (Status)

Accepted - 2026-06-02

## 컨텍스트 (Context)

운영자가 Post 본문에 이미지를 넣을 수 있어야 한다. 이는 PRD의 기존 범위 외("이미지·미디어 업로드 제외")와 ADR-0003(본문 마크다운 한정)의 가정을 확장하는 결정이다. 지금은 단일 서버 로컬 디스크로 충분하지만, 추후 S3 등 오브젝트 스토리지로 옮길 수 있어야 한다. 본문은 마크다운(ADR-0003)이므로 이미지는 본문과 어떻게 연결되는지도 정해야 한다.

평가 기준: 저장 백엔드 교체 용이성, ADR-0003 마크다운 본문과의 정합성, 보안(업로드 검증·XSS).

## 결정 (Decision)

이미지 업로드를 **저장소 추상화 인터페이스(StorageProvider)** 뒤에 구현한다. MVP는 `LocalStorageProvider`(서버 로컬 디스크)로 구현하고, 인터페이스 교체만으로 `S3StorageProvider`로 확장할 수 있게 한다.

- 업로드는 운영자 전용 엔드포인트(`POST /api/uploads`, JWT 가드)로 받고, 저장 후 **접근 URL을 반환**한다.
- 운영자는 반환된 URL을 본문 마크다운에 `![](url)`로 임베드한다(별도 첨부 엔티티를 만들지 않음).
- 렌더 시 `rehype-sanitize`(ADR-0003)는 안전한 `img` 태그/`src`를 허용하도록 구성한다.
- 파일은 타입 화이트리스트(이미지 MIME)와 크기 상한으로 검증한다.
- 저장 위치·공개 URL 베이스는 환경변수로 주입한다(로컬 경로 / 향후 S3 버킷).

## 결과 (Consequences)

긍정:
- 저장 백엔드를 인터페이스 교체만으로 로컬 → S3로 확장할 수 있다.
- 이미지를 마크다운 URL로 임베드해 ADR-0003 본문 모델을 그대로 유지하고 스키마 변경이 최소화된다.

부정/감수해야 할 것:
- 로컬 저장은 단일 서버에 종속되어 수평 확장·컨테이너 재배포 시 영속 볼륨이 필요하다.
- 마크다운 임베드 방식은 어떤 이미지가 어느 Post에 쓰였는지 추적이 약해, 미사용 파일 정리(가비지 컬렉션)가 별도 과제로 남는다.
- 업로드 검증(타입·크기·악성 파일)과 sanitize의 img 허용 정책을 신중히 설정해야 한다.

## 검토 시점

다중 인스턴스 배포, 이미지 트래픽 증가, 또는 미사용 파일 정리가 필요해지면 S3StorageProvider 전환과 이미지-Post 연관 추적(첨부 엔티티) 도입을 재평가한다.
