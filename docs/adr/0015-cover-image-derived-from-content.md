# ADR-0015: 목록 요약·대표 이미지를 저장 컬럼 없이 본문에서 파생

## 상태 (Status)

Proposed - 2026-06-04

## 컨텍스트 (Context)

글 목록(`GET /api/posts`) 카드에 각 글의 **대표 이미지**(glossary)와 읽기 좋은 **요약**을 노출해야 한다. Post 본문은 마크다운 단일 원문으로 저장된다(ADR-0003). 대표 이미지는 "본문 첫 이미지"로 정의했고, 요약도 본문에서 만든다. 이 두 값을 어떻게 만들고 어디에 둘지 정해야 한다.

평가 기준: ADR-0003(마크다운 본문 단일 원천)과의 정합성, 스키마/마이그레이션 비용, 목록 조회 성능, 본문 수정 시 일관성 유지 비용.

## 결정 (Decision)

대표 이미지와 평문 요약을 **별도로 저장하지 않고, 목록 조회 시 본문(`contentMarkdown`)에서 파생**한다.

- `PostService.toSummary()` 가 `extractFirstImageUrl(contentMarkdown)` 로 본문 첫 이미지(마크다운 `![](url)` 또는 원시 `<img src>` 중 먼저 나오는 것)를 뽑아 `PostSummaryDto.coverImageUrl`(없으면 `null`)로 노출한다.
- 같은 메서드가 `toSummaryText(contentMarkdown)` 로 이미지/코드/링크/헤딩/강조 표기를 제거한 평문 요약을 만든다.
- 목록 쿼리는 이미 본문을 포함(`include` 관계 조회)하므로 **추가 DB 조회·컬럼·마이그레이션이 없다**.
- 파생 로직은 순수 함수(`cover-image.ts`, `markdown-summary.ts`)로 두어 단위 테스트한다.

## 결과 (Consequences)

긍정:
- 본문이 단일 진실원으로 유지된다(ADR-0003). 본문을 고치면 대표 이미지·요약이 자동으로 따라온다 — 동기화 코드가 필요 없다.
- 스키마 변경·마이그레이션·백필이 없다. 변경 표면이 작다.
- 목록 쿼리가 이미 본문을 들고 있어 성능 추가 비용은 정규식 추출(목록 페이지 크기만큼)뿐이다.

부정/감수해야 할 것:
- 매 목록 조회마다 본문 파싱이 일어난다(현재 저볼륨 1인 블로그라 무시 가능, 페이지 크기 10).
- 대표 이미지를 **운영자가 수동 지정할 수 없다**(첫 이미지로 고정). 본문에 이미지가 없으면 대표 이미지도 없다.
- 목록 응답이 본문을 메모리에서 다루므로, 본문이 매우 커지면 추출 비용이 커진다.

## 검토 시점

목록 트래픽·본문 크기가 커져 파싱 비용이 문제되거나, 대표 이미지 수동 지정/크롭 요구가 생기면 `coverImageUrl` 영속 컬럼(작성·수정 시 계산·저장) 또는 대표 이미지 지정 UI 도입을 재평가한다.
