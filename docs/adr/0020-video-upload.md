# ADR-0020: 본문 비디오 업로드 (이미지 업로드 확장, ffmpeg 없이)

## 상태 (Status)

Proposed - 2026-06-04

## 컨텍스트 (Context)

운영자/작성자가 글 본문에 짧은 데모 클립을 넣으려면 지금은 외부 호스팅에 의존해야 한다.
이미지 업로드(ADR-0012)와 같은 UX 로 본문에 비디오를 임베드할 수 있어야 한다. 동시에 다음
제약이 있다:

- 단일 서버 로컬 디스크 환경(ADR-0012 의 LocalStorageProvider)을 유지한다.
- 컨테이너 이미지에 ffmpeg 등 무거운 의존을 추가하고 싶지 않다.
- 본문 마크다운 모델(ADR-0003)과 sanitize 정책을 가능한 한 건드리지 않는다.
- 권한 모델(ADR-0018) 은 그대로다 — 업로드는 AUTHOR/ADMIN 만.

## 결정 (Decision)

이미지 업로드(ADR-0012)의 패턴을 그대로 확장해 **MP4 비디오를 같은 엔드포인트로** 받는다.

- **엔드포인트**: `POST /api/uploads` 단일 — 이미지/비디오를 MIME 화이트리스트로 분기.
  분리 엔드포인트(`/uploads/video`)는 만들지 않는다.
- **포맷**: `image/jpeg|png|gif|webp` + `video/mp4` 1종. WebM/MOV 는 범위 외.
- **크기 상한**: env 분리. `UPLOAD_MAX_BYTES=5242880`(이미지 5MB), `UPLOAD_MAX_BYTES_VIDEO=52428800`(비디오 50MB).
- **응답**: 기존 모양 유지하고 `type: 'image' | 'video'` 한 필드만 추가. 클라이언트 분기 보조.
- **본문 표기**: 마크다운은 변경 없이 `![alt](url)` 그대로. 렌더러가 URL 확장자가 `.mp4` 면
  `<video controls preload="metadata" playsInline>` 로 치환. 새 syntax 도입 안 함.
- **목록 썸네일**: 서버 썸네일 추출(ffmpeg) 하지 않는다. 클라이언트에서
  `<video preload="metadata" muted playsInline>` 의 첫 프레임을 그대로 카드 커버로 사용.
- **정적 서빙**: express static 이 자동으로 `Accept-Ranges: bytes` + `Range` 요청을
  `206 Partial Content` 로 응답하므로 별도 코드 없음.

`PostSummaryDto.coverImageUrl` 의 이름은 그대로 유지한다(마이그레이션 없음). 값이 `.mp4` 면
클라이언트가 비디오로 그린다. 호환 우선 — 후속 정리 시 `coverMediaUrl` 로 이름 통일을 재평가.

## 결과 (Consequences)

긍정:
- 서버 의존성 0 추가. 작성자 UX 가 이미지와 동일(마크다운/응답 모양 동일).
- 마크다운 본문/스키마/Domain Event 모두 변경 없음(ADR-0003·0012 의 모델 그대로).
- 클라이언트 첫 프레임 썸네일이 모든 MP4 에 균일하게 동작 — 별도 데이터 모델 불필요.

부정/감수해야 할 것:
- 카드의 첫 프레임 표시는 메타데이터 + 첫 프레임 분량의 다운로드를 매번 트리거한다(목록 1
  스크롤에서 비디오 N개면 N회). 모바일 데이터 사용량/대역폭이 증가할 수 있다 — 후속 모니터링.
- 같은 비디오의 시킹 요청은 다중 Range 호출을 만든다. 단일 서버 트래픽이 늘면 CDN/오브젝트
  스토리지로 전환할 때까지 부담을 안고 간다(ADR-0012 의 S3 전환 트리거와 통합).
- 확장자(`.mp4`)에 의존하는 렌더 분기는 확장자 위조에 약하지 않게 서버 MIME 검증과 한 쌍으로
  관리해야 한다(둘 다 통과해야 저장됨).
- `MOV/WebM` 미지원으로 iPhone 원본을 그대로 못 올리는 UX 제약. 후속 ADR 에서 확장 검토.

## 검토 시점

목록 트래픽이 늘어 대역폭이 문제가 되거나, 작성자들이 모바일 원본(MOV)을 변환 없이 올리고
싶다는 요청이 누적되면 (a) 서버 ffmpeg 썸네일/트랜스코딩, (b) 오브젝트 스토리지 + CDN,
(c) 포맷 확장(WebM/MOV) 을 재평가한다.
