# 평가자 루브릭 (Evaluator Rubric) — my-blog

> 에이전트의 *세션 산출물* 품질을 6차원으로 채점하는 점수표.
> "에이전트가 이번 세션에서 잘했는가?" 에 답한다. (코드베이스 자체의 건전성은 `docs/quality-document.md`.)
> 템플릿 출처/비교: `docs/harness-template-comparison.md`.

## 언제 채점하나
- `/finish {ID}` 의 verifier 패스 직후(태스크 1건 단위). → 아래 "/finish 연동" 참고.
- 기능 1종 완결(마일스톤) 후, 또는 "이번 작업 평가해줘" 요청 시.

## 채점 방식
- 6차원 각 **0 / 1 / 2점** (만점 12).
- **저자가 자기 점수를 매기지 않는다.** verifier/code-reviewer 가 모은 *증거*에 근거해 채점한다(저자 편향 차단).
- 점수만 적지 말고 **증거 인용**(테스트 출력·curl·리뷰 지적)을 1줄 붙인다.

## 6차원

| # | 차원 | 0점 | 1점 | 2점 |
|---|---|---|---|---|
| 1 | **정확성 (Correctness)** | acceptance 미충족/오동작 | 핵심 동작은 맞으나 엣지 누락 | acceptance 전 항목이 목표 동작과 일치(verifier PASS) |
| 2 | **검증 (Verification)** | 검증 안 돌림/증거 없음 | 일부만 실행(예: 단위만) | lint+typecheck+test(+e2e) 통과 **+ 증거 기록**. 파일 반환 기능은 쓰기-읽기 왕복(규칙 #9) |
| 3 | **범위 규율 (Scope)** | 태스크 밖 파일 다수 변경 | 약간의 범위 이탈 | 해당 태스크 ID 범위 안에서만 변경(한 번에 한 태스크 — 규칙 #2) |
| 4 | **신뢰성 (Reliability)** | 재실행/재시작 시 깨짐 | 환경 의존 잔존 | 재실행해도 GREEN 유지, 테스트 DB 격리(규칙 #8) 준수 |
| 5 | **유지보수성 (Maintainability)** | 의도 불명·주석 없음 | 부분적으로만 명료 | 다음 세션이 이해 가능(한글 주석, ADR/handoff로 결정 추적) |
| 6 | **핸드오프 준비성 (Handoff)** | handoff 없음/공란 | 일부 섹션 누락 | `docs/handoff/{날짜}-{ID}.md` 6섹션 완비, feature_list↔tasks 동기화(규칙 #10) |

## 결론

- **Accept (≥10, 0점 항목 없음)** — 기준 충족. `/finish` 진행.
- **Revise (6–9, 또는 0점 1개 이하)** — 수락 전 수정. 무엇을 고치면 2점인지 명시.
- **Block (≤5, 또는 1·2번 차원에 0점)** — 근본 문제. status 는 todo 유지, 멈춤.

> 정확성(1)·검증(2)에 0점이면 총점과 무관하게 **Block**. 이 둘은 "됐다"의 핵심이라 가중한다.

## /finish 연동

`/finish {ID}` 절차에 **7.5단계(채점)** 가 추가됐다(`.claude/commands/finish.md`):
2단계 verifier 가 acceptance 를 PASS/FAIL/UNVERIFIED 로 판정한 뒤, 그 증거로 이 6차원을 채점한다.
결론이 **Block** 이면 commit 하지 않고 멈춘다. **Revise** 면 사용자에게 수정 항목을 보고한다.
채점 결과 한 줄은 handoff 노트 하단에 `평가: Accept/Revise/Block (n/12)` 로 남긴다.

## 보정 (Calibration) — 중요

기본 설정의 에이전트는 자기 평가에 관대하다(문제를 보고도 통과로 논리 전개). 다음을 반복해 인간 판단과 정렬한다:

1. 완료된 태스크 묶음에 이 루브릭을 적용한다.
2. 점수를 사람 판단과 비교한다.
3. 어긋난 곳에서 각 차원의 0/1/2 기준을 더 구체화한다(이 표를 수정).
4. 다시 적용해 정렬을 확인한다.
5. 일관 일치까지 3–5라운드. 각 라운드 변경은 아래 로그에 남긴다.
