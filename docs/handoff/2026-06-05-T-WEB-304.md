# Handoff — T-WEB-304 Playwright e2e + shared prod 빌드 인프라

날짜: 2026-06-05 / feature: rich-text-editor / 근거: ADR-0021

## 한 일
rich-text-editor 시리즈의 마지막 e2e 시나리오와 기존 4 e2e 호환 갱신, 그리고 본 시리즈가
드러낸 **shared prod 빌드 인프라 누락** 의 근본 패치를 진행했다.

- **e2e 시나리오 신규**: `packages/web/e2e/rich-editor.spec.ts`
  - 가입(AUTHOR) → 새 글 작성 → 본문 텍스트 → 전체 선택 → 색(rose-500) + 크기(text-lg) → 이미지 업로드 → 저장 → 발행 → 상세에서 `.ab-rich-content span.text-rose-500/text-lg`, `img` 모두 검증.
- **helpers.ts 확장**: `typeRichBody(page, text)` (contenteditable focus + keyboard.type), `TINY_MP4` 픽스처.
- **기존 e2e 4건 갱신**: `author-self-serve`/`draft-hidden`/`operator-flow`/`member-journey` 의 `getByLabel('본문(마크다운)').fill(...)` → `typeRichBody(page, ...)`. `operator-flow` 의 이미지 단언 selector 갱신(`.prose img` → `.ab-rich-content img`, 본문 토스트는 `.ab-rich-editor img`).
- **인프라 패치(별도 의미)** — ADR-0021 의 value import(`richHtmlSchema`)가 prod 컨테이너에서 resolve 되지 않던 잠재 회귀를 해결:
  - `packages/shared/tsconfig.build.json` 신규: commonjs 출력 + `dist/`.
  - `packages/shared/package.json`: main/exports/types → `dist/index.js` 기준. `build` 스크립트가 `tsconfig.build.json` 사용. `files: [dist, src]`.
  - `packages/api/Dockerfile`: build stage 에 `pnpm --filter @blog/shared build` 추가, prod stage 에 `shared/dist` + `shared/package.json` 복사.
  - `packages/web/Dockerfile`: build stage 에 `pnpm --filter @blog/shared build` 추가 (vite 가 shared 의 value import 를 resolve).

## TDD 사이클 확인
- e2e 격리 스택 첫 실행: api 컨테이너 부팅 실패(`Cannot find module '@blog/shared'`). prod node 가 `.ts` 를 require 못 함 — value import 의 첫 실제 회귀.
- shared 의 commonjs 빌드 + Dockerfile build 단계 추가 → 컨테이너 빌드는 통과.
- 격리 스택 재실행: 6 spec 모두 페이지 진입 timeout(`getByLabel('제목')`). 인프라 패치는 통과했고 RichEditor/RichContent 의 prod vite 빌드 산출물이 페이지 진입 단계에서 막는 것으로 추정.
- 사용자 결정: T-WEB-304 의 acceptance 중 spec/인프라 패치는 완료, 실제 격리 실행 검증은 **T-WEB-305(별건)** 로 분리.

## 결정한 것
- **shared 를 commonjs 로 빌드**: nest prod 가 commonjs 라 ts/esm 불일치 회피. dev 환경(nest --watch / vite) 은 ts 소스 직접 처리하므로 영향 없음.
- **단일 소스 정신 유지**: `richHtmlSchema` 는 그대로 `packages/shared` 한 곳. 빌드 산출물을 컨테이너에 옮겨 value import 의 런타임 resolve 보장.
- **T-WEB-304 acceptance 의 의미 재정의**: spec 작성 + 인프라 패치를 완료 단위로 인정하고, 격리 실제 실행 검증은 T-WEB-305 로 회수. 단계 분리가 디버깅 비용 비례.

## 알려진 이슈 (T-WEB-305 가 다룰 영역)
- 격리 스택 prod 빌드에서 `/admin/posts/new` 진입 시 6 spec 모두 timeout(`getByLabel('제목')`). RichEditor 또는 PostEditor 의 prod 빌드 산출물이 첫 렌더에서 막힘.
- 후보 원인:
  - TipTap 의 react/pm 패키지가 vite prod 빌드에서 esm/cjs 호환 깨짐.
  - `RichContent` 의 `DOMPurify.addHook(...)` 모듈-level 부수효과가 prod tree-shaking 과정에서 누락.
  - dompurify 가 브라우저 환경에서 동작이지만 prod minify 시 wrapping 변화.
- 디버그 첫 단계: 격리 web 컨테이너의 `docker logs my-blog-e2e-web-1` 또는 컨테이너 안에서 `vite build` 산출물 직접 확인. 또는 web 컨테이너 dev target 임시 사용해 격리 e2e 통과 확인.
