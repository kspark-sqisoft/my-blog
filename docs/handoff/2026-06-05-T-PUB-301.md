# Handoff — T-PUB-301 PostService contentHtml 입력 + 서버 sanitize

날짜: 2026-06-05 / feature: rich-text-editor / 근거: ADR-0021

## 한 일
사용자가 실제로 보내는 `contentHtml` 입력에 서버 sanitize 게이트를 적용했다. T-INFRA-303 가
만든 `convertMarkdownToHtml` / `sanitizeRichHtml` 헬퍼를 서비스 계층이 재사용해, 입력 종류와
관계없이 응답·DB 의 contentHtml 은 항상 sanitize 통과한 결과만 들어간다.

- DTO:
  - `CreatePostDto.contentMarkdown` 을 optional 로 완화 + `contentHtml?: string` 추가.
  - `UpdatePostDto.contentHtml?: string` 추가.
- `PostService`:
  - `resolveBody(input)` 헬퍼 — contentHtml 우선 `sanitizeRichHtml`, 없으면 `convertMarkdownToHtml(contentMarkdown)`. 둘 다 비면 `BadRequestException`.
  - `create()` 가 resolveBody 결과를 `{ contentMarkdown, contentHtml }` 양쪽에 저장(호환 유지).
  - `update()` 는 부분 수정 — contentHtml 가 들어오면 그쪽 우선 sanitize, contentMarkdown 만 오면 변환·sanitize 양쪽 동기. body 변경 없으면 DB 도 그대로.
  - `PostWithTags` 타입과 `toDetail()` 응답에 `contentHtml` 추가.
- `PostController.create` 가 `dto.contentHtml` 도 서비스에 전달.
- e2e 3 신규(`test/post.e2e-spec.ts`):
  - contentHtml 입력 → 응답에 contentHtml(sanitize 결과) 포함.
  - `<script>`/`onerror` 입력 → 응답·상세 GET 양쪽에서 제거(서버 sanitize 게이트).
  - contentMarkdown 만 보내는 기존 호출도 호환(서버가 변환해 contentHtml 채움).

## 결정한 것
- **양방향 입력 허용**: contentMarkdown 만 보내는 기존 클라이언트 호환을 위해 서버가 마크다운 변환을 책임진다. 클라이언트는 점진적으로 contentHtml 로 전환(T-WEB-302/303).
- **응답에 양쪽 노출**: 과도기 동안 응답에 contentMarkdown 와 contentHtml 모두 포함. T-WEB-303 후 별도 ADR/태스크에서 contentMarkdown 응답 제거(acceptance 3 의 후반부).
- **서버 sanitize 가 최종 게이트**: 클라이언트 sanitize(T-WEB-303 의 dompurify) 는 추가 방어이고, 보안 책임은 서버에 있다.

## 알려진 이슈
- 응답에 contentMarkdown / contentHtml 두 키가 보인다 — 검색·인덱싱 시 중복 처리 주의. 후속 정리.
- contentHtml 가 빈 문자열인 글(과거 데이터)은 T-INFRA-303 마이그레이션으로 모두 채워졌다고 가정. 빈 contentHtml 가 있으면 응답에도 빈 문자열.
- `update()` 의 bodyPatch 가 둘 다 미지정이면 DB 의 contentHtml 은 stale 한 contentMarkdown 의 변환 결과로 유지(이전 값 그대로 — 정상). 별도 동기 정책 필요시 후속.
