# Handoff — T-PUB-203 비디오 정적 서빙 Range 응답 e2e

날짜: 2026-06-04 / feature: video-upload / 근거: ADR-0020

## 한 일
업로드된 MP4 가 `<video>` 의 seek/일시정지·재개에 필요한 `Accept-Ranges: bytes` 헤더와
`Range` 요청에 대한 `206 Partial Content` 응답을 정상 돌려주는지 명시적 e2e 회귀 가드를 추가했다.

- `test/upload.e2e-spec.ts` 2 신규 케이스:
  - "업로드한 MP4 는 video/mp4 + Accept-Ranges 로 서빙된다 (왕복 + 절대규칙 #9)"
  - "Range: bytes=0-99 → 206 Partial Content + 100 바이트"
- 코드 변경 0 — `app-setup.ts:19` 의 `useStaticAssets`(express.static) 가 기본 동작으로
  `Accept-Ranges`/`Range`/`206` 을 처리해 RED 없이 GREEN 통과. 향후 정적 서빙 미들웨어 교체 또는
  prefix 변경 시 이 가드가 회귀를 잡는다.

## TDD 사이클 확인
- RED 의도: e2e 5개 케이스 중 비디오 정적 서빙·Range 가 깨지면 잡히게.
- 실제 RED: 발생하지 않음 — express.static 기본 동작이 이미 충족. T-PUB-202 시점부터 잠재적으로 통과 가능했으나 가드가 없었음.
- GREEN: 별도 구현 없이 통과. 회귀 가드로서 가치 — 후속 변경(예: NestJS S3 어댑터 또는 다른 static 미들웨어로 교체)이 이 행동을 깨면 즉시 검출.

## 결정한 것
- 정적 서빙은 express.static 그대로 사용. Range 처리 직접 구현 없이 미들웨어에 위임.
- 회귀 가드 e2e 는 절대 규칙 #9(쓰기-읽기 왕복) 의 비디오 버전으로 자리 잡음. 이미지 왕복 케이스는 이미 존재.

## 알려진 이슈
- `useStaticAssets` 이 처리하지 못하는 매우 큰 비디오의 다중 Range(multipart byteranges, `bytes=0-99,200-299`) 케이스는 검증하지 않았다 — `<video>` 가 보통 단일 Range 만 보내므로 영향 작음.
- T-PUB-202 의 multer `limits` 미설정 이슈는 그대로(별건).
