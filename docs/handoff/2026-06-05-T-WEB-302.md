# Handoff — T-WEB-302 PostEditor 통합 (textarea → RichEditor)

날짜: 2026-06-05 / feature: rich-text-editor / 근거: ADR-0021

## 한 일
PostEditor 의 본문 입력을 마크다운 textarea 에서 TipTap 기반 `<RichEditor>` 로 교체했다.
사이드바의 별도 "미디어" 패널을 제거하고 미디어 업로드는 RichEditor 도구바로 통합했다.

- `pages/admin/PostEditor.tsx` 전체 재작성:
  - 본문 입력: `<textarea>` → `<RichEditor>`. `body` 상태가 HTML 문자열로 의미 변경.
  - 저장 payload: `{ title, contentHtml: body, tags }` (contentMarkdown 입력 안 보냄).
  - 빈 본문 검증: `isBlankHtmlBody(html)` 헬퍼 — 태그/공백/`&nbsp;` 제거 후 길이 0 검사.
  - 수정 모드 로드: `existing.contentHtml || existing.contentMarkdown` 폴백(과도기).
  - 미디어 업로드 콜백: 클라이언트 화이트리스트(`ALLOWED_UPLOAD_MIME`) → `useUploadImage.mutateAsync` → `{ url, type }` 형태로 RichEditor 에 전달.
  - 사이드바 "미디어" 패널 제거(도구바 통합) — 작성 흐름이 본문 내에서 끝남.
- `pages/admin/PostEditor.test.tsx` 재작성:
  - `RichEditor` 를 `vi.mock` 으로 textarea + 미디어 input 형태로 mock. RichEditor 본동작은 별도 spec(8 케이스).
  - 9 케이스: 저장 payload(contentHtml), 수정 모드 contentHtml 로드 + PATCH, contentHtml 빈 경우 contentMarkdown 폴백, 빈 본문 차단, 제목 검증·포커스, 이미지/MP4 업로드, PDF 차단, 태그 6개 차단.

## TDD 사이클 확인
- RED: 9 케이스 작성 → 1 fail (수정 모드 contentMarkdown 폴백). 원인은 `existing.contentHtml ?? existing.contentMarkdown` 의 `??` 가 빈 문자열을 폴백 안 시킴.
- GREEN: `??` → `||` 변경. 9/9 통과.
- 의도된 사이드 효과: 응답에 contentHtml 빈 상태인 과도기 row 에서도 작성자가 기존 마크다운을 그대로 본다.

## 결정한 것
- **응답 양쪽 노출 의존**: 백엔드 응답에 contentHtml 과 contentMarkdown 둘 다 있으므로 클라이언트가 폴백 가능. 후속 ADR 에서 contentMarkdown 응답 제거 시 폴백 라인도 같이 정리.
- **빈 본문 검증은 클라이언트 단계만**: HTML 정규식 제거로 단순화. 서버 검증(PostService.resolveBody) 이 최종 게이트라 안전.
- **사이드바 미디어 패널 제거**: 작성 흐름이 본문 내에서 완결. 도구바의 미디어 업로드 input(`accept="image/*,video/mp4"`) 한 곳.
- **spec mock 전략**: RichEditor 를 textarea 로 mock 해 양식·저장 흐름만 검증. 실제 에디터 동작은 RichEditor.test 가 책임. 통합 e2e 는 T-WEB-304 가 책임.

## 알려진 이슈
- 시각 스타일(`.ab-rich-editor`/`.ab-rich-toolbar` 클래스의 CSS) 는 아직 없음 — T-WEB-303 또는 별도 보강에서 `index.css` 에 토큰 추가 필요. 기능 영향 없음.
- 사용자 시연 시점은 dev 컨테이너의 web hot reload 가 의존성 변경(@tiptap 추가)을 반영하려면 `--renew-anon-volumes` 가 필요할 수 있다(CLAUDE.md 함정).
