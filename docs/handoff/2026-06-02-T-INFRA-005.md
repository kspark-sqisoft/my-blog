## 한 일

Playwright E2E 가 그동안 공유 dev DB(`blog`, 5433)에 직접 붙어 dev 데이터를 오염시킬 수 있던
구조를 끊고, **전용 격리 스택(`my-blog-e2e`)** 으로 분리. jest 통합 테스트(`blog_test`)에 이미
적용된 격리 원칙을 Playwright 트랙에도 동일하게 적용해 **3-tier DB 격리**(blog · blog_test ·
blog_e2e)를 완성.

### 변경 파일

- **`docker-compose.e2e.yml`** (신규) — 자체완결 격리 스택.
  - `db`: postgres:17-alpine, `blog_e2e`, 호스트 5434:5432, `tmpfs` (매 실행 새 데이터)
  - `api`: prod target, 호스트 3002:3000, `DATABASE_URL` → `db:5432/blog_e2e`, `UPLOAD_DIR=/tmp/e2e-uploads`
  - `web`: prod target(nginx), 호스트 5174:80, nginx 가 `/api`·`/uploads` 를 `api:3000` 프록시
  - 모든 서비스 healthcheck. 별도 프로젝트 이름 `my-blog-e2e` 로 dev 컨테이너/네트워크 충돌 0.
- **`scripts/e2e-isolated.sh`** (신규, 실행권한) — 한 명령으로 다음 단계 자동:
  1. Playwright 브라우저 확인(필요 시 chromium 다운로드, idempotent)
  2. 격리 db up + healthy 대기 (`compose up --wait db`)
  3. 호스트에서 `prisma migrate deploy` → `blog_e2e`
  4. 호스트에서 `OPERATOR_*` 시드
  5. 격리 api + web up + healthy 대기
  6. `playwright test` 실행 (E2E_BASE_URL=http://127.0.0.1:5174)
  7. dev DB(`blog`)가 떠 있으면 실행 전/후 `(posts:comments)` 카운트 비교
  - `EXIT` 시 자동 `docker compose down -v`. `E2E_KEEP_STACK=1` 로 디버깅용 보존 가능.
- **`packages/web/package.json`** — `test:e2e` 를 격리 스크립트 호출로 변경.
  `test:e2e:raw` 는 기존 `playwright test` (외부에서 격리 스택을 띄운 경우용).
- **`.env.example`** — `E2E_DB_PORT/E2E_API_PORT/E2E_WEB_PORT/E2E_POSTGRES_*/E2E_OPERATOR_*/E2E_JWT_SECRET/E2E_BASE_URL/E2E_KEEP_STACK/E2E_COMPOSE_PROJECT` 문서화.
- **`docs/harness.md`** — "운영 함정 #5" 추가: 3-tier DB 격리, 격리 회귀 가드, 디버깅 옵션, prod CMD 부수 수정 기록.
- **`feature_list.json`** — T-INFRA-005 등록 후 done.

### 부수 발견 + 동시 수정 — prod 빌드가 절대 부팅 안 되던 버그

- **`packages/api/Dockerfile`** — prod target `CMD ["node", "dist/main"]` → `["node", "dist/src/main"]`.
- 원인: `nest build` 산출 경로가 tsconfig.build 기준 `dist/src/main.js` 인데 CMD 가 `dist/main` 을 가리켜
  `Cannot find module '/repo/packages/api/dist/main'` 으로 즉시 종료.
- 격리 스택 첫 부팅이 이 버그를 표면화 → 운영 첫 배포가 동일하게 깨질 뻔한 사전 보증.
