# Handoff — T-PUB-202 /api/uploads MP4 수락 + MIME/크기 분리 검증

날짜: 2026-06-04 / feature: video-upload / 근거: ADR-0020

## 한 일
`POST /api/uploads` 엔드포인트를 단일 그대로 두면서 `video/mp4` 도 수락하도록 확장했다.
T-PUB-201 에서 환경변수와 응답 모양을 준비해 둔 위에 controller 본체를 확장한 변경.

- `upload.controller.ts`:
  - `ALLOWED_MIME` 에 `video/mp4` 추가 (이미지 4종 + 비디오 1종).
  - `imageFileFilter` → `mediaFileFilter` 로 리네이밍, 메시지 갱신("이미지 또는 MP4 비디오").
  - `resolveMediaType(mimetype)` 헬퍼 추가 — `'video' | 'image'` 분기.
  - 크기 한도 분기: 비디오는 `UPLOAD_MAX_BYTES_VIDEO`(env / fallback 50MB), 이미지는 `UPLOAD_MAX_BYTES`(env / fallback 5MB). 응답 모양 그대로 + `type` 필드만 동적.
- `test/upload.e2e-spec.ts`:
  - `UPLOAD_MAX_BYTES_VIDEO=2048` (테스트용 분리 한도 — 이미지 1KB / 비디오 2KB).
  - 5 신규 케이스 — MP4 201+type, 비디오 한도 초과 413, **이미지 한도 1KB 초과 1.5KB MP4 가 허용되는 MIME 분리 회귀 가드**, MOV 400, MEMBER 강등 후 403.
  - 기존 성공 응답 단언에 `type:'image'` 회귀 가드 추가.

## 결정한 것
- multer `limits.fileSize` 는 설정하지 않고 컨트롤러 후검증으로만 분리 한도 적용 — multer 미설정 시 50MB 정도는 메모리에 들어와도 부담 없다는 가정. 후속 압박 시 streaming/object storage 전환 트리거(ADR-0012/0020 검토 시점).
- 응답 모양은 유지(`{url, contentType, size, type}`) — 클라이언트는 `type` 또는 url 확장자 어느 쪽으로 분기해도 일관.
- 확장자 위조 차단은 MIME 검증으로 끝. 추후 magic-byte 검사가 필요해지면 후속 ADR.

## 알려진 이슈
- multer `limits.fileSize` 부재로 매우 큰 파일(예: GB 단위)이 메모리에 올라올 위험. dev/소형 운영엔 OK. 부하 모니터링 필요.
- 50MB 비디오의 응답 본문 + 동기 디스크 쓰기 — 단일 노드에서 동시 요청이 많아지면 latency 영향.
