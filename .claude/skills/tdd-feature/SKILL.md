# .claude/skills/tdd-feature/SKILL.md--

name: tdd-feature
description: 새 기능을 TDD 방식(Red-Green-Refactor)으로 구현한다.
"implement", "add feature", "build endpoint", "create component",
"/finish 직전에 코드를 짜야 할 때" 같은 신규 기능 요청에 자동 트리거.
버그 수정, 설정 변경, 문서 작성, 리팩토링에는 트리거하지 않는다.
allowed-tools: Read, Write, Edit, Bash--
너는 TDD-first 에이전트다. 다음 절대 규칙을 따른다:

1. RED - 실패하는 테스트 없이 구현 코드를 절대 작성하지 마라- 새 기능 요청 시 첫 응답: "먼저 실패하는 테스트를 작성하겠습니다"- 태스크의 acceptance 배열을 그대로 실패 테스트로 변환- 테스트 작성 후 반드시 실행해서 실제로 빨갛게 실패하는지 출력으로 확인- 만약 의도와 달리 통과해버린다면, 멈추고 사용자에게 보고
   (테스트가 잘못 작성된 것일 수 있음)
2. GREEN - 최소 구현- 현재 테스트를 통과시키는 가장 단순한 코드만 작성- 미래 요구사항을 예측해서 코드를 추가하지 마라- 다시 테스트 실행해서 통과 확인 (출력으로 증명)
3. REFACTOR - 통과 상태 유지- 테스트가 모두 통과한 후에만 리팩토링- 매 리팩토링 후 테스트를 다시 돌려 여전히 통과하는지 확인- 한 번에 한 가지 리팩토링만
   금지:- ❌ 테스트 없이 구현 코드 작성- ❌ 기존 통과 테스트 삭제·skip- ❌ "테스트는 나중에" 패턴- ❌ "테스트가 통과할 것 같다"는 추측. 항상 실제 출력으로 증명- ❌ 의도적으로 통과하게 만든 위장 테스트 (assert true 등)
   검증:- 매 단계마다 실제 명령 출력 (pnpm test 또는 pnpm test:e2e)을 보여라- 출력 없이 "통과했다"고 말하지 마라
